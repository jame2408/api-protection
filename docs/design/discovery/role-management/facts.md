# Facts — role-management discovery

> Phase A grilling 事實登記簿。每輪只登記使用者親口確認的事實（標注輪次）；AFK 蒐集的 repo 事實獨立成段並附出處。未決點一律進未決段，不代答。

## 初始輸入（原文）

- 使用者提問（2026-07-11，session 內前一輪）：「我們有新增角色，譬如 admin，請問有需求可以管理這些角色，以及這些角色可以使用哪些 API 的設定嗎？」
- `/domain-discovery` 派工引數：「開場探索角色管理需求」。

## AFK 蒐集的 repo 事實（探索前既有狀態，非使用者作答）

- 角色現為封閉集合：api-spec §2.1 定義 PlatformAdmin／SecurityAdmin／TenantAdmin／Consumer＋內部 System；角色來自 JWT `role` claim，由發 token 方決定，本服務不發行、不儲存、不管理角色（ADR-024）。
- 「角色→API」映射現為兩層寫死：文件層 api-spec §2.1 角色-路徑前綴表；執行層 endpoint authorization policy（現僅 `SuspendKeyEndpoint` 落了 `RequireRole`，其餘為 ADR-024 後置債）。
- `AccessPolicy` BC 管理的是 API key 的 scope（Data Plane），非操作者角色權限（Control Plane）——兩個授權層次。
- 46 個 BDD 場景中無任何角色管理場景；Discovery 新場景產出凍結中（解凍屬使用者裁決）。
- design-doc §14 Q6（開放問題，未裁決）：「三種外部角色與系統內部實體的對應關係為何？權限模型採 RBAC 還是 Claim-based？」——認證側已由 ADR-024 以 claim-based 落地；角色管理（role CRUD／角色-API 指派）無任何既往 ADR 裁決或拒絕。

## 使用者確認的事實（逐輪增補）

- **R1**：使用者確認其想的「角色管理」與五種操作者角色（Control Plane）**不是同一個使用場景**；其心中場景為「我有多個 API 提供給 A 團隊使用」——即「哪個團隊可以使用哪些（業務）API」的存取管理。探索方向由「操作者角色管理」轉向待確認的「API 消費方存取管理」。

- **R2**：使用者確認「多個 API」指 **Data Plane 層**——A 團隊拿金鑰呼叫的業務 API，即 Scope／AccessPolicy 這一層；非 Control Plane 管理 API。探索範圍就此定錨於「Data Plane 的消費方存取管理」。

- **R3**：規模與痛點——現有 **3 個團隊、約 10 支 API**；痛點為**新團隊接入時要逐把金鑰設 scope 很麻煩**（onboarding 場景的重複性設定成本）。

- **R4**：每團隊持有 **2–3 把金鑰，依環境分開（測試、正式）**。規模盤點：3 團隊 × 2–3 把 × 約 10 支 API——onboarding 一個新團隊＝對 2–3 把金鑰重複設定同一組 scope。

- **R5**：變動頻率——**約一季一至兩次**，以**新 API 上線開放給既有團隊**為最常見情境。

- **R6**：權限集合變動的生效方式——使用者裁決**重新發放金鑰**（金鑰在建立當下複製權限快照；集合變動不回寫既有金鑰）。與 design-doc §5「scope 移除不自動修改既有金鑰」的既有立場一致，無張力。

- **R7**：實體對應確認——**一個團隊＝一個 Consumer**（2–3 把金鑰皆掛其名下，依環境分）。R2-open 銷案。

- **R8**：治理結構——**去中心化、以 API 提供方為單位**：A 團隊管理 A 團隊自己的服務，並**由 A 團隊發 key 給其他團隊**使用其 API；同時 A 團隊也持有其他團隊發的 key（雙面角色：provider 兼 consumer）。無單一中央管理者統管所有存取。此事實修正 R7 的理解：「一個團隊＝一個 Consumer」只描述了團隊的**使用方面向**；團隊同時具有 Service Owner（提供方）面向。

- **R9**：授權流程——**使用方（B 團隊）申請、提供方（A 團隊）核准**。存取的建立是一個申請→核准的工作流，非提供方單方面推送。

- **R10**：租戶結構裁決——**選項一：全公司單一租戶，「團隊」成為租戶內的一等模型概念**（含各團隊對自己服務的管理權界線）。裁決背景：選項二（每隊一租戶）會使跨團隊發 key 撞上「租戶間互不往來」的核心不變量。R8-open-a 銷案。

- **R11**：核准的產物——**核准建立持久的「授權關係」（資格），金鑰在該資格下另行建立**（測試／正式各一把）；重發金鑰（R6）在同一資格下進行，無須重新申請。「授權」與「金鑰」是兩個獨立生命週期。

- **R12**：授權關係的結束——實務上**收回幾乎只因資安議題發生**（如外洩、異常使用）；無定期到期／審查機制，也無主動歸還的實務慣例。授權關係預設長存，結束是例外事件而非常態流程。

- **R13**：使用者確認無其他重要事實，Phase A 收攏（2026-07-11）。
- **Phase B hotspot 裁決（2026-07-11）**：H2＝修改既有授權（仍需核准）；H3＝級聯撤銷；H4＝團隊管理者制（隊內角色分層）；H6＝授權不分環境（環境屬金鑰層）。H1 留 open question；H5 交 Phase C。
- **Phase C 裁決（2026-07-11）**：使用者整包接受模型建議——`Team`＋`AccessGrant` 兩新 aggregate、H5＝一實體兩面向（Consumer 為 Team 使用方投影 1:1）、新 BC **Team Access** 分類 Supporting、KeyLifecycle 微擴、Scope Registry 歸屬調整。
