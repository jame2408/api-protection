# 企業級 API 金鑰管理系統：需求探索與領域建模

## 1. 執行摘要

API 是企業對外開放服務的主要方式。當內部微服務或資料開放給第三方、外部開發者或其他業務單元使用時，身份驗證與授權就是第一道關卡。

在機器對機器（M2M）的自動化通訊中，API 金鑰因為整合成本低、效能高，仍是業界主流做法。但 API 金鑰本質上是「持有者令牌」（Bearer Token）——拿到字串就能存取。一旦金鑰洩漏，攻擊者可無限制存取敏感資料，直到金鑰被撤銷。

因此，這份報告的目標不只是「產生隨機字串」，而是設計一套**全生命週期的金鑰治理機制**，在開發者體驗（DX）的流暢性與資安管控之間取得平衡。

報告聚焦在**需求探索**與**領域建模**，不涉及具體技術實作（程式語言、資料庫、雲端供應商等）。從威脅模型出發，推導系統的核心實體、屬性、狀態機邏輯與安全合規需求。

## 2. 背景與威脅模型

### 2.1 從邊界安全到零信任

傳統企業安全依賴網路邊界（Perimeter Security），假設「防火牆內即安全」。但 API 開放給外部後，請求可能來自網際網路任何地方。API 金鑰管理系統必須採用零信任（Zero Trust）原則：

- 不信任任何網路位置
- 不信任任何未經加密的憑證
- 假設網路隨時可能被監聽

這帶出首要架構原則：**金鑰管理系統必須獨立於業務邏輯之外**。業務服務不應自行處理金鑰驗證邏輯，而應依賴統一的閘道器或邊車（Sidecar）模式，由管理系統提供驗證與授權決策。

### 2.2 威脅模型（Threat Modeling）

| 威脅向量 | 攻擊描述 | 防禦需求 |
|:---|:---|:---|
| 金鑰洩漏 (Key Leakage) | 開發者不慎將金鑰提交至公開程式碼儲存庫或包含在前端程式碼中 | R-SEC-01: 金鑰包含可識別前綴，便於自動掃描工具識別並撤銷。R-SEC-02: 支援公開儲存庫掃描整合 |
| 彩虹表攻擊 | 攻擊者取得金鑰資料庫備份，嘗試逆向還原金鑰 | R-SEC-03: 禁止明文或可逆加密儲存。必須使用加鹽的高強度雜湊演算法（如 Argon2id） |
| 重放攻擊 (Replay Attack) | 攻擊者攔截合法 API 請求並重複發送 | R-SEC-04: 強制 HTTPS/TLS 傳輸加密。R-SEC-05: 支援請求簽章（Request Signing）作為高級驗證模式 |
| 權限提升 | 使用低權限金鑰（如唯讀）嘗試存取高敏感介面（如寫入） | R-GOV-01: 金鑰必須綁定精細化的權限範圍（Scopes），每次請求嚴格校驗 |
| 資源耗盡 (DDoS/Abuse) | 利用合法金鑰發送海量請求，導致後端服務癱瘓 | R-GOV-02: 每個金鑰關聯速率限制（Rate Limit）與配額（Quota）策略 |
| 側信道攻擊 (Timing Attack) | 透過分析驗證回應的時間差，推測金鑰有效性或內容 | R-SEC-06: 驗證邏輯必須採用恆定時間（Constant-time）比較演算法 |

## 3. 系統角色與使用者畫像

### 3.1 API 消費者（API Consumer / Developer）

通常是合作夥伴的開發工程師或內部其他團隊的開發者。

- **需求**：整合效率與服務穩定性。金鑰即刻生效，輪替時不影響線上服務。
- **安全意識**：通常較低。傾向申請最大權限，可能將金鑰硬編碼在應用程式中。系統必須透過強制性流程（強制過期、最小權限預設值）來降低風險。

### 3.2 服務負責人（API Provider / Service Owner）

開放 API 的業務團隊負責人。

- **需求**：知道誰在使用 API、使用了多少流量，能在緊急時阻斷特定使用者的存取。
- **職責**：定義 API 的權限範圍（Scopes）與流量配額（Plans）。

### 3.3 安全管理員（Security Admin / CISO Office）

負責企業整體資安合規。

- **需求**：所有操作有據可查（Audit Logs），系統符合 PCI-DSS 或 GDPR 等法規要求。
- **特權**：擁有全局視角，可執行緊急撤銷，覆蓋服務負責人的設定。

### 3.4 系統代理（System Agent / Automation）

非人類角色，代表背景執行的自動化程序。負責掃描過期金鑰、發送輪替通知、執行異常檢測、清理無效歷史資料。

## 4. 概念資料建模

### 4.1 核心實體：API 金鑰憑證

API 金鑰不只是一個字串，它是一個帶有豐富上下文的物件。

| 屬性 | 類型 | 說明 | 安全原則 |
|:---|:---|:---|:---|
| Key ID | UUID | 在日誌、UI、API 呼叫中引用的公開識別碼。絕不可用金鑰本身作為 ID | 資訊洩露防護 |
| Key Prefix | String (7-10 chars) | 金鑰明文的前綴（如 `sk_live_`），用於識別金鑰類型（測試/生產）與所屬服務 | 識別性與防呆 |
| Key Hash | String (Hex/B64) | 金鑰明文經加鹽雜湊後的密文。禁止儲存明文 | 縱深防禦 |
| Truncated Key | String (Last 4 chars) | 金鑰後四碼，用於管理後台顯示（如 `...a9B3`），讓開發者能區分不同金鑰 | 可用性與安全性平衡 |
| Consumer ID | Foreign Key | 關聯至申請該金鑰的專案或組織 | 責任歸屬 |
| Environment | Enum | 金鑰適用環境（Sandbox / Production）。測試金鑰禁止在生產環境生效 | 環境隔離 |
| Scopes | Array of Strings | 授權的操作權限列表（如 `orders:read`, `refunds:create`） | 最小權限原則 |
| Lifecycle Status | Enum | 當前狀態：Active, Rotating, Revoked, Expired, Locked | 狀態機管控 |
| Created At | Timestamp | 金鑰生成時間 | 審計追蹤 |
| Expires At | Timestamp | 金鑰失效時間點。系統不應允許「永久有效」的金鑰 | 週期性輪替 |
| Last Used At | Timestamp | 最近一次成功驗證的時間，用於識別「殭屍金鑰」 | 清理與維護 |

### 4.2 實體：存取策略（Access Policy）

將安全規則從金鑰實體中剝離，形成獨立的策略物件。

**IP 白名單（IP Allowlist）**

- 一組 CIDR 網段（如 `192.168.1.0/24`）
- 支援 IPv4 與 IPv6、單一 IP 與網段
- 若設定了白名單，來自非清單內 IP 的請求即使金鑰正確也必須拒絕

**速率限制策略（Rate Limit Policy）**

- `quota_limit`：週期內總次數（如 100 萬次/月）
- `rate_limit`：瞬時速率（如 50 RPS）
- `burst`：突發容忍度
- 超過配額應返回 HTTP 429 錯誤

### 4.3 實體：審計日誌（Audit Log）

每一筆針對金鑰的管理操作都必須生成一條不可篡改的記錄。

| 屬性 | 說明 |
|:---|:---|
| Event ID | 唯一識別碼 |
| Actor | 執行操作的使用者 ID 或系統服務名稱 |
| Action | 操作類型（Create, Revoke, Rotate, Update Policy） |
| Resource | 受影響的 Key ID |
| Snapshot Before | 修改前的屬性快照（JSON） |
| Snapshot After | 修改後的屬性快照（JSON） |
| Reason | 操作理由（如「懷疑洩漏」），撤銷操作尤其需要 |
| Context | 來源 IP、User-Agent 等環境資訊 |

## 5. 核心功能需求：全生命週期管理

將金鑰的生命週期定義為有限狀態機（FSM）。

### 5.1 金鑰生成（Generation）

**R-GEN-01: 高熵值生成**
使用密碼學安全的偽亂數產生器（CSPRNG）。金鑰長度至少 256 位元（32 bytes），建議採用 Base62 編碼確保 URL 安全性，最終字串長度約 40-60 字元。

**R-GEN-02: 格式化結構**
金鑰應遵循 `[Prefix]_[RandomPart]_[Checksum]` 格式：
- Prefix：如 `acme_prod_`，用於識別
- Checksum（選用）：末尾附加校驗碼，讓後端在不查詢資料庫的情況下快速過濾格式錯誤的請求

**R-GEN-03: 單次顯示原則（Display Once）**
生成時只回傳一次完整的金鑰明文。前端必須明確警告使用者「金鑰只會顯示這一次，請立即保存」。對話框關閉後清除記憶體中的明文，資料庫中僅存雜湊值，連管理員也無法找回。

### 5.2 安全儲存（Secure Storage）

**R-STR-01: 單向雜湊**
- 禁止儲存金鑰明文
- 禁止使用對稱加密（如 AES）儲存金鑰——一旦主金鑰洩漏，所有 API 金鑰都會淪陷
- 必須使用專為密碼儲存設計的慢速雜湊演算法（如 Argon2id 或 PBKDF2-SHA256），能抵禦 GPU 暴力破解與彩虹表攻擊
- 高熵機器 token 豁免（docs/adr/adr-017-key-hash-hmac-and-hotpath-contract.md）：系統生成且隨機熵 ≥ 128-bit 的 API 金鑰，以 HMAC-SHA256 + server-side pepper 儲存，不適用慢速雜湊要求；本豁免不及於任何人類選擇的秘密（密碼、自訂 token）

**R-STR-02: 獨立加鹽**
每個金鑰的雜湊運算必須使用隨機生成的獨立鹽值（Salt），防止攻擊者預先計算雜湊表。高熵機器 token 依 ADR-017 以 server-side pepper 取代 per-key salt：128-bit 隨機輸入本身即不可預計算，而確定性雜湊是驗證熱路徑唯一索引查找的必要條件。

### 5.3 驗證與授權（Validation & Authorization）

最高頻的運作場景，對效能與安全性都有高要求。

**R-VAL-01: 恆定時間比較**
驗證雜湊值時必須使用恆定時間比較函式，防止側信道計時攻擊。無論輸入金鑰在第幾位字元錯誤，驗證函式的執行時間都應一致。

**R-VAL-02: 多層級過濾**
驗證邏輯遵循漏斗模型以節省資源：

1. 格式檢查：檢查前綴與長度（記憶體操作，極快）
2. 狀態檢查：從快取查 Key ID 狀態是否為 Active
3. IP 檢查：比對來源 IP 是否在白名單內
4. 雜湊驗證：執行雜湊運算比對
5. 權限檢查：確認金鑰是否擁有存取該 Endpoint 的 Scope

**R-VAL-03: 快取策略**
雜湊運算昂貴且資料庫讀取慢，所以必須在 API Gateway 層實施快取。

- 快取內容：`KeyHash -> Metadata (Scopes, IP, Status)`
- 風險：快取會造成撤銷延遲。必須實施**主動快取失效**（Active Cache Invalidation）：金鑰被撤銷或修改時，立即廣播訊息清除各 Gateway 節點的對應快取

### 5.4 輪替與過渡（Rotation & Transition）

最容易導致服務中斷的環節，需要精細的狀態管理。

**R-ROT-01: 零停機輪替**
支援「雙活狀態」（Dual Active State）。使用者觸發輪替時，系統生成新金鑰（Key B），並將舊金鑰（Key A）標記為 Rotating。

**R-ROT-02: 寬限期（Grace Period）**
在可配置的寬限期內（如 24 小時或 7 天），Key A 與 Key B 同時有效，讓開發者有時間將客戶端更新為 Key B。寬限期結束後，系統自動將 Key A 轉為 Revoked。

**R-ROT-03: 強制輪替策略**
高敏感服務應支援強制輪替（如每 90 天）。若開發者未在期限內輪替，系統發送多次警告，最終自動失效金鑰。

### 5.5 撤銷與緊急處置（Revocation）

安全事件發生時，速度就是一切。

**R-REV-01: 立即阻斷**
管理員擁有「緊急按鈕」，能基於 Key ID 或 Consumer ID 立即撤銷所有存取權。該操作具最高優先級，繞過所有快取機制（如透過全局 Pub/Sub 事件通知所有節點）。

**R-REV-02: 撤銷原因記錄**
撤銷操作必須強制填寫原因（如「金鑰洩漏」、「員工離職」、「專案結束」），供後續安全審計與威脅分析使用。

## 6. 安全治理與合規性需求

### 6.1 精細化權限控制

**R-GOV-03: 範圍限制**
- 金鑰不預設擁有「管理員」或「全功能」權限
- 系統定義一組標準化的權限字串（如 `user:read`, `order:write`）
- 建立金鑰時，強制要求開發者選擇最小可用權限集
- UI 要提示過度授權的風險

**R-GOV-04: 環境隔離**
生產環境的金鑰禁止調用測試環境的 API，反之亦然。防止開發過程中的誤操作影響生產資料。

### 6.2 監控與警報

**R-MON-01: 使用量異常檢測**
系統建立每個金鑰的流量基線（Baseline）。流量異常激增（如 5 分鐘內增加 1000%）時觸發警報，可選擇自動暫時鎖定金鑰。

**R-MON-02: 異常來源檢測（Impossible Travel）**
若同一個金鑰在極短時間內從地理位置相距很遠的兩個 IP 發起請求（如 5 分鐘內分別來自台北和紐約），應視為金鑰被盜用的信號。

### 6.3 合規性審計

**R-AUD-01: 不可否認性**
所有管理操作日誌必須包含足夠的上下文資訊，確保操作者無法否認其行為。日誌應匯出至外部的 SIEM 系統或冷儲存，並設置 WORM（一次寫入，多次讀取）策略以防篡改。

## 7. 運維監控與異常檢測

### 7.1 即時儀表板

服務負責人需要回答以下問題：

- 當前最大的 API 使用者是誰？
- 是否有金鑰即將過期？
- 是否有金鑰頻繁觸發 403 Forbidden 錯誤？（可能代表配置錯誤或攻擊嘗試）

### 7.2 自動化防禦規則

系統應支援配置自動化規則（Rule Engine）：

- 規則 A：若金鑰 K 在 1 分鐘內觸發超過 50 次 401/403 錯誤 → 自動轉為 Locked 並發送 Slack 通知給管理員
- 規則 B：若金鑰 K 來自未知的 ASN（如某個惡名昭彰的 VPN 提供商）→ 自動降級其速率限制

## 8. 使用者介面與開發者體驗

### 8.1 開發者入口網（Developer Portal）

- **自助服務**：開發者自行生成、輪替與命名金鑰，無需人工審批（除非涉及高敏感權限）
- **整合指引**：顯示金鑰時，動態生成 `curl` 範例，示範如何正確將金鑰放入 HTTP Header（如 `Authorization: Bearer <Key>`），避免開發者將金鑰放入 URL 參數（易被 Proxy 紀錄檔記錄）
- **安全提示**：在介面上顯著位置提醒「不要將此金鑰提交到 Git」、「建議使用環境變數」

### 8.2 管理後台（Admin Console）

- **全局視圖**：管理員能搜尋、過濾所有金鑰
- **一鍵處置**：針對可疑金鑰提供「暫停（Suspend）」與「撤銷（Revoke）」功能。暫停是可逆的，用於調查期間；撤銷是永久的

---

## 9. 結論

這份報告定義了企業級 API 金鑰管理系統的需求模型，涵蓋零信任架構、全生命週期管理、雙活輪替機制與審計日誌。四個核心原則：

1. **金鑰即密碼**：必須以儲存使用者密碼的同等規格（雜湊、加鹽）來儲存 API 金鑰
2. **狀態機是核心**：嚴謹定義金鑰的狀態流轉（特別是輪替期間的寬限期）是保障業務連續性的關鍵
3. **治理重於防禦**：防火牆是基礎，但基於身份的治理（權限範圍、配額、審計）才是應對 M2M 威脅的有效手段
4. **體驗即安全**：流暢的輪替工具與清晰的錯誤提示能降低開發者犯錯的機率

此需求規格書應作為後續系統架構設計與技術選型的基礎文件。
